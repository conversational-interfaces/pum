This specification defines the syntax and semantics for utility model's that are used to create and drive conversational experiences. Utility Models allow authors to encode preferences over states of the world; enabling languages such as DMPL to autonomously drive agents towards settings with the highest possible utility.

Conformance

As well as sections marked as non-normative, all authoring guidelines, diagrams, examples, and notes in this specification are non-normative. Everything else in this specification is normative.

Introduction

Scope

This section is informative.

This document is limited to specifying the syntax and semantics of designing and incorporating utility into any task-oriented programming language. One example application of Utility Models arises in chat-bot settings where authors are able to specify preferences between interactions allowing for a more fluid and dynamic conversation.

Utility Model Format

Utility interface

This section is normative. 
        interface Utility {
          attribute Pairs preferences;
        };

preferences attribute

The preferences attribute specifies a Pairs object denoting a ranking configuration of all provided examples. It is specified in a key:value manner where key always takes on the value utility and value is a Config of pairings. 

          "utility": "[Pair]"

Pairs interface

        interface Pairs {
          attribute Config configs;
        };

configs attribute

The configs attribute specifies an tuple of Config objects. For the sake of simplicity we assume that the first Config object has higher utlity value than the second Config object. 

            "configs": "[firstConfig, secondConfig]"

Config interface

        interface Config {
          attribute String rankings;
        };

rankings attribute

The rankings attribute specifies a list containing any number of examples corresponding to desired outcomes. As mentioned in the Pairs interface, the first specified item of each list is assumed to have higher user preferences then its corresponding pair. 

            [
              [{numWrongAttempts: 1, success: true}, {numWrongAttempts: 0, success: false}],
              [{numWrongAttempts: 5, success: true}, {numWrongAttempts: 3, success: false}],
            ]

Name

A name is a sequence of characters conforming to those in the XML standard. 

	                name ::= [ http://www.w3.org/TR/xml-names/#NT-NCName ]

String

A string literal represents a finite sequence of characters. It is surrounded by starting and ending quotation marks. 

	                string :: = '"' name '"'

Examples

This section is informative.

In a setting where a client is autonomously attempting to effectively teach a user a, for example mathematical concept, clients are able to specify the preferences of the states. In this case we've specified that a higher count of wrong attempts is preferred over a lower count, as this shows the user is at least attempting to learn. 

        [
          [{numWrongAttempts: 1, success: true}, {numWrongAttempts: 0, success: false}],
          [{numWrongAttempts: 5, success: true}, {numWrongAttempts: 3, success: false}],
        ]

Being able to assign utility allows us to indirectly specify a desired trajectory in which animating all characters in a game setting has higher preferences than beginning the game.

                "`start_game`": true
              },
              {
                "`is_pig_visible`": false,
                "`start_game`": true
              }
            ],
            [

                "`is_frog_visible`": true,

              {
                "`is_fox_visible`": true,
                "`start_game`": true
              },
              {
                "`is_fox_visible`": false,
                "`start_game`": true
              }
            ]
        ]
